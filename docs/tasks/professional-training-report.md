# Task: Professional AI-Generated Training Report on the Recommendations Page

> **Status:** Proposed
> **Area:** Recommendations / AI Coach
> **Type:** Feature (cross-cutting: backend insights, AI prompts, frontend report UI, export)
> **Owner:** Unassigned

---

## 1. One-Line Summary

Upgrade the **Recommendations** page from a list of short coaching phrases into a **professional, statistics-rich training report** — generated by AI over the runner's **full Strava history**, with detailed workload analytics, pace/effort/splits breakdowns, and a **timeline of recommended training** for the next block (not just one-liners).

---

## 2. Problem Statement

Today the Recommendations page (`frontend/src/pages/Recommendations.jsx`) and its backing endpoints (`GET /api/recommendations` and `GET /api/recommendations/coach-review` in `backend/routes/recommendations.js`) give the runner:

- A small set of recommendation cards (`title`, `description`, `reasoning`, `actionItems`) built from rule-based heuristics in `backend/services/mlService.js`.
- A 28-day window of summary metrics, a weekly trend, and a short "weekly summary" headline from OpenAI in `backend/services/openaiCoach.js`.
- A race pace prediction limited to the **last 7 days** in `buildRacePacePrediction` (see `backend/services/trainingInsights.js`).

Limitations the user has called out:

1. **Not enough statistical workload data.** Athletes want load metrics (CTL/ATL/ACWR, monotony, strain), pace-zone distribution, HR-zone time-in-zone, split-by-split analysis, week-over-week deltas, and seasonal context — not just totals.
2. **Pace / effort / split insights are missing or shallow.** `backend/services/activityInsights.js` already has the seed of a per-run split insight, but it is not surfaced in the recommendations report, and there is no AI-generated narrative tying splits to effort and progression across activities.
3. **Only the last N days feed recommendations.** The route queries `Activity.find({ userId, date: { $gte: sinceDate } })`. The product needs to ingest **the full record of Strava activities** to spot multi-month patterns, seasonal regressions, and long-term load trends.
4. **Output is too phrase-y.** Recommendations are short cards. The user wants a **professional report**: structured sections, charts, a **time-lapse / timeline** of recommended sessions for the next block, and an exportable artifact (PDF / shareable view).

---

## 3. Goal

Deliver a **Professional Training Report** experience accessible from the Recommendations page that:

- Pulls **the runner's entire Strava activity history** (with sensible caps and incremental refresh) into the analytics pipeline.
- Computes a deep set of **workload, pace, effort, and split** statistics, including time-series over weeks and months.
- Uses **AI (OpenAI, with a graceful rule-based fallback)** to generate a narrative report grounded in those statistics — multi-section, professional in tone, citing specific numbers.
- Produces a **time-lapse / week-by-week training timeline** for the next 4–12 weeks with specific session prescriptions (type, distance, pace target, duration, effort cue), not just generic phrases.
- Is **exportable** (in-app printable / PDF view) so the runner can share or save the report.

---

## 4. In Scope

### 4.1 Backend — analytics pipeline

Create `backend/services/trainingReport.js` (new) that orchestrates a richer analytics pipeline. It must accept the runner's **entire `Activity` history** (not a `days` window) and produce a single `TrainingReport` payload:

- **Workload metrics**
  - Weekly distance, duration, elevation, run count for **every ISO week with activity in the last 365 days** (so the report can render a long horizontal time-lapse, not just 6 weeks).
  - Acute (7-day), chronic (28-day), and seasonal (90-day) rolling distance and duration.
  - **Acute/Chronic Workload Ratio (ACWR)** time series (extend `backend/services/loadRisk.js` to expose the rolling series, not just current value).
  - **Training monotony** (mean / std of daily load over a 7-day window) and **training strain** (monotony × weekly load), week by week.
  - Rest-day distribution: number of days off in each 7-day window.
- **Pace analytics**
  - Distribution of activity pace across **pace zones** (Easy, Steady, Threshold, Interval, Repetition) derived from each activity's `pace` field and the user's `goalPaceMinPerKm` (fallback: percentiles of the runner's own historical pace).
  - Per-week average / median / fastest pace.
  - **Negative-split rate**, **positive-split rate**, **even-pace rate** across all activities that have `splits_metric` or `streamSummary.paceMinPerKm` data, reusing the split logic in `backend/services/activityInsights.js`.
- **Effort analytics**
  - Heart-rate zone time-in-zone (Z1–Z5) using `streamSummary.heartrate` when available and `User.age` (or stored max HR) for the zone bands.
  - Aggregate **Strava `suffer_score`** per week if available on the synced detail.
  - HR-decoupling estimate per long run (first-half vs second-half HR per unit pace) where stream data exists.
- **Split analytics**
  - For each activity with km splits, store `{ fastestKm, slowestKm, deltaSecPerKm, pacingPattern }`.
  - Aggregate **pacing pattern frequency** across all runs (e.g. "63% of long runs were positive splits in the last 12 weeks").
- **Personal records and progression**
  - Best-effort time per common distance bucket (1 km, 5 km, 10 km, 15 km, 21.1 km, 42.2 km) from each activity's `movingTime` & `distance`, with the date they were set.
  - 6-month progression line for each of the above buckets.
- All numerical outputs **must be rounded** for stable rendering and labelled with units; no `NaN` / `Infinity` should escape the service.

### 4.2 Backend — full-history ingestion

- Add a query path that selects `Activity.find({ userId })` with **no `date` lower bound**, plus a hard cap (e.g. last 1,000 activities, ordered desc) to bound memory.
- Index check: `backend/models/Activity.js` already declares `{ userId: 1, date: -1 }`; verify it covers the new query and add a compound index if necessary.
- For per-run split / stream analytics, only operate on data already persisted to MongoDB (do **not** issue a per-activity Strava API call inside the report endpoint — that would blow the rate limit). If `streamSummary` is missing, the report degrades gracefully and notes which runs lack data.

### 4.3 Backend — AI report generation

Extend `backend/services/openaiCoach.js` with `generateTrainingReport({ user, history, analytics, raceContext })` that:

- Sends a compact, **statistics-only JSON payload** (not raw activities) to OpenAI so the prompt stays small and deterministic.
- Uses `response_format: { type: 'json_object' }` and a strict schema:

```json
{
  "executiveSummary": "string (3-5 sentences)",
  "workloadAssessment": {
    "headline": "string",
    "narrative": "string (paragraph)",
    "callouts": ["string", "string"]
  },
  "paceAndEffortAnalysis": {
    "headline": "string",
    "narrative": "string",
    "zoneCommentary": "string"
  },
  "splitAndPacingAnalysis": {
    "headline": "string",
    "narrative": "string"
  },
  "progressionAndPRs": {
    "headline": "string",
    "narrative": "string"
  },
  "risksAndWatchouts": ["string", "string"],
  "nextBlockPlan": {
    "blockGoal": "string",
    "durationWeeks": 4,
    "weeks": [
      {
        "weekNumber": 1,
        "focus": "string",
        "targetDistanceKm": 35,
        "sessions": [
          {
            "day": "Mon",
            "type": "Easy",
            "distanceKm": 6,
            "targetPaceMinPerKm": 6.4,
            "durationMinutes": 38,
            "effortCue": "Conversational; nose-breathing"
          }
        ],
        "keySession": "string",
        "longRun": { "distanceKm": 18, "targetPaceMinPerKm": 6.0 },
        "recoveryNotes": "string"
      }
    ]
  },
  "raceReadiness": {
    "verdict": "ready | build | rebuild | recover | taper",
    "explanation": "string",
    "predictedFinishTimeLabel": "string"
  }
}
```

- Must validate every key exists and **fall back to a rule-based `buildFallbackTrainingReport(...)`** that fills the same schema deterministically when `OPENAI_API_KEY` is unset or the call fails.
- Reuse the existing OpenAI client construction pattern in `openaiCoach.js`; keep `temperature` at `0.3` for stability.
- Honour the same `process.env.OPENAI_MODEL` override, defaulting to `gpt-4o-mini`.

### 4.4 Backend — new endpoint

Add to `backend/routes/recommendations.js`:

```
GET /api/recommendations/professional-report
  Query: raceDistance?, raceDate?, raceName?, planWeeks? (default 6, max 12)
  Auth:  required (same `auth` middleware as the other routes)
  Response: { success, analyzedActivities, generatedAt, analytics, aiReport, source: 'openai' | 'rules' }
```

- The handler must:
  1. Load the user.
  2. Pull the full activity history (with the 1,000-row cap above).
  3. Call the new `trainingReport.js` analytics builder.
  4. Call `generateTrainingReport(...)`.
  5. Persist a lightweight pointer (e.g. `lastReportGeneratedAt` on the user, or a new `TrainingReport` model — author's choice; pick the lower-risk option and justify in the PR description).
  6. Return the combined payload.
- Track usage with the existing `UsageEvent` pattern so admin analytics can see adoption.
- Apply the same rate-limit middleware that already wraps `/api/recommendations`.

Add a client method in `frontend/src/services/api.js`:

```js
recommendationsApi.getProfessionalReport = ({ raceDistance, raceDate, raceName, planWeeks } = {}) =>
  api.get('/recommendations/professional-report', {
    params: { raceDistance, raceDate, raceName, planWeeks }
  });
```

### 4.5 Frontend — Professional Report UI

On `frontend/src/pages/Recommendations.jsx`, add a new section (collapsible card or dedicated tab) titled **"Professional Training Report"** with:

- A "Generate professional report" button (loading state, disabled while generating).
- An offline-friendly cache using the existing `loadSnapshot` / `saveSnapshot` helpers, keyed `professional-training-report`.
- Report layout (when data is present):
  1. **Cover header** — runner name, report date, analyzed activity count, history span ("X activities over the last Y months"), next race chip if set.
  2. **Executive summary** — paragraph from `aiReport.executiveSummary`.
  3. **Workload section**
     - Stat tiles: weekly average km, peak week, ACWR (latest), monotony, strain, rest days/week.
     - Long-horizon **time-lapse chart** of weekly distance and duration for up to 52 weeks (reuse `TrainingTrendChart` styling or extend it; allow horizontal scroll on small screens).
     - ACWR sparkline over the last 12 weeks with safe-band shading (0.8–1.3 ok, >1.5 elevated).
     - AI narrative from `aiReport.workloadAssessment`.
  4. **Pace & effort section**
     - Pace-zone distribution bar / donut.
     - HR-zone time-in-zone bar (only when data exists; otherwise a "Connect HR data for deeper insights" note).
     - AI narrative from `aiReport.paceAndEffortAnalysis`.
  5. **Splits & pacing section**
     - Table or chips for negative / even / positive split rates.
     - AI narrative from `aiReport.splitAndPacingAnalysis`.
  6. **Progression & PRs section**
     - Sortable table of distance buckets with current best and date.
     - 6-month line chart of one selectable distance bucket.
     - AI narrative from `aiReport.progressionAndPRs`.
  7. **Risks / watchouts** — bulleted list from `aiReport.risksAndWatchouts` rendered with `WarningIcon`.
  8. **Next block timeline (time-lapse)** — the key new view:
     - Horizontal stepper / timeline with one column per week from `aiReport.nextBlockPlan.weeks`.
     - Each week column shows: week number, focus, target distance, key session, long run, recovery notes.
     - Each session row inside a week shows day, type, distance, target pace, duration, effort cue.
     - Selecting a week opens a detail drawer with the full session list and an "Add to calendar" placeholder (`.ics` export is **optional** stretch).
  9. **Race readiness** — verdict chip + paragraph + predicted finish.
  10. **Export controls** — "Print / Save as PDF" button using `window.print()` plus a print-only stylesheet that hides nav/footer and keeps section page breaks clean.
- Loading skeletons and an `Alert` for the rules-based fallback ("AI was unavailable; this report uses rule-based analytics.").
- Respect the existing dark/light theme (MUI tokens, no hard-coded colours).

## 5. Out of Scope (for this task)

- Sharing the report via a public URL or email digest.
- Calendar `.ics` export (mark as stretch, fine to leave as a follow-up issue).
- Coach-to-athlete multi-tenant features.
- Mobile push notifications for the report.
- Recomputing or backfilling `streamSummary` for old activities — the report should degrade gracefully when streams are absent.

## 6. Data & Architectural Notes

- **Source of truth.** All analytics consume the `Activity` collection. No new Strava API calls in the report path.
- **Performance budget.** The endpoint should respond in **< 5 s** for a user with 1,000 activities. Profile if needed; avoid per-activity DB round trips, use a single `find()` and operate in memory.
- **Privacy.** Do not send raw GPS polylines or per-second streams to OpenAI. Only send aggregated statistics. The fallback path must produce a usable report with **no** external network calls.
- **Determinism.** Round all stats before sending to the model; the same input twice should produce a similar narrative. Save the analytics payload alongside any persisted report so the UI can re-render without re-prompting the model.
- **Backwards compatibility.** Do not change the existing `GET /api/recommendations` or `GET /api/recommendations/coach-review` response shapes; the new report is an additive endpoint.

## 7. Acceptance Criteria

A reviewer should be able to verify each of the following:

- [ ] `GET /api/recommendations/professional-report` returns the full schema described in §4.3 for a user with at least 5 historical activities.
- [ ] With `OPENAI_API_KEY` unset, the same endpoint returns a complete, well-formed report with `source: "rules"`.
- [ ] The analytics payload includes ACWR series, monotony, strain, pace-zone distribution, HR-zone time-in-zone (when HR data exists), split-pattern counts, and per-distance PRs.
- [ ] The Recommendations page renders the new "Professional Training Report" section with all subsections from §4.5 and the next-block timeline (time-lapse) view.
- [ ] The report renders an offline snapshot from `localStorage` when the backend is unreachable, with a visible "saved at" timestamp.
- [ ] `window.print()` produces a clean, paginated PDF with no navigation chrome and no overflow clipping in the timeline.
- [ ] No regressions: existing recommendation cards, coach review, weekly summary, training progress, vector insights, and similar-segment panels still render and refresh as before.
- [ ] No `Activity.findById` / per-activity Strava call inside the report path; the endpoint completes in under 5 seconds on a profile with 1,000 activities (measure locally and note the timing in the PR description).
- [ ] All new numeric outputs are finite and rounded; no `NaN` / `Infinity` appears in the JSON.
- [ ] Lint passes for backend (`cd backend && npm test` if a lint/test script exists) and frontend builds (`cd frontend && npm run build`).

## 10. Non-Functional Requirements

- **Accessibility:** every chart needs an accessible text fallback (screen-reader-friendly summary or `aria-label`).
- **Responsive:** the timeline must scroll horizontally on screens narrower than ~900 px without breaking the rest of the page.
- **Theming:** all colours must come from the MUI theme; respect the existing dark mode toggle.
- **Telemetry:** track a `professional_report_view` and `professional_report_generated` usage event (extend `ALLOWED_CLIENT_USAGE_EVENTS` in `backend/routes/coach.js` if the client emits them).
- **Security:** the endpoint requires the existing `auth` middleware; never accept an arbitrary `userId` from the query.

---

## 11. Risks & Open Questions

- **Strava data completeness.** Most of the deeper insights (HR zones, splits, streams) depend on data already being on the activity. Decide whether to lazily backfill `streamSummary` for the report — recommended: **no**, but expose a "Sync details" CTA next to runs missing stream data.
- **Model cost.** Generating on every page load is wasteful. Suggested approach: cache the last report on the user (or in a new collection) and only regenerate on explicit button click or when new activities arrive.
- **Edge cases.** Brand-new users (≤ 2 activities) should see a friendly empty state instead of a partial report.
- **Storage of past reports.** Single-report-per-user vs. history? Default to overwriting the latest unless the team wants a history view.

---

## 12. Done = …

When a runner with months of synced Strava activity opens the Recommendations page, clicks **Generate professional report**, and within a few seconds receives a multi-section, statistics-rich, AI-narrated report ending in a week-by-week training timeline they can scroll through and print, with the same experience degrading gracefully when AI or HR data is unavailable.
